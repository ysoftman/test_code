스타일링 용어 설명

헷갈리는 이유: "runtime" 이 두 축에서 쓰입니다. ① CSS 를 언제 만드나(생성 시점) ② 브라우저에서 스타일용 JS 가 도나(런타임 비용).

CSS-in-JS

CSS 를 별도 .css 파일이 아니라 JS/TS 코드 안에서 작성하는 방식. 라이브러리가 클래스명을 자동 생성·스코프 격리해 줍니다. (예: MUI Emotion, styled-components, Panda, vanilla-extract)

유틸리티 CSS

flex, px-4 같은 작은 단위 클래스를 조합해 스타일링. 빌드 때 사용한 클래스만 CSS 로 생성. CSS-in-JS 가 아닙니다. (예: Tailwind, shadcn)

컴포넌트 라이브러리

버튼·카드 등 완성된 컴포넌트를 제공하는 묶음. (예: MUI, Chakra UI) — 내부 스타일 엔진은 별개(MUI=Emotion runtime, Chakra v3=Panda 기반).

① 생성 시점 — build-time vs runtime

구분 CSS 를 만드는 시점 결과
build-time 개발자가 빌드(bun run build)할 때 정적 .css 로 미리 추출
runtime 사용자가 페이지를 열어 브라우저에서 JS 실행될 때 JS 가 <style> 을 동적 주입

② 런타임 비용 — zero runtime vs runtime

zero runtime: 브라우저에서 스타일 생성을 위해 도는 JS 가 0. 보통 build-time 의 결과입니다(미리 다 구워둠 → 브라우저는 완성된 CSS 만 로드). runtime: 스타일 엔진 JS 가 브라우저에서 동작 → 약간의 비용, 대신 props 기반 동적 스타일에 유연.

비유 — zero runtime: 미리 구워둔 빵(바로 가져감) · runtime: 주문 시 굽는 빵(유연하지만 매번 작업).
